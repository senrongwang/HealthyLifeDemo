# 健康生活（HealthyLife）主干实施方案

> 本方案基于 `docs/设计文档.md`，将整体实现拆分为有序阶段与可独立交付的任务。
> 配合 `待完成功能列表.md` 跟踪进度：每完成一项任务后更新其状态并在该文件追加完成记录。
>
> **修订说明（截图核对）**：本方案已对照 `screenshots/healthyfile1-8.png` 实际参考界面修订。主要调整：
> 1. MainTab 由 3 Tab 改为 **2 Tab（首页/我的）**，成就改为从"我的"菜单进入的**独立页面 AchievementPage**；
> 2. 添加/编辑任务由 openCustomDialog 弹窗改为**独立页面 TaskAddPage / TaskEditPage**（含路由跳转）；
> 3. 打卡改为点击任务项弹出**带插画的 ClockInDialog 弹窗**（openCustomDialog），非列表内直接打卡；
> 4. WeekCalendar 改为**徽章式**（日期行 + 星期标签 + 六边形徽章 + 左右切换箭头）；
> 5. TaskListItem 移除独立打卡按钮，右侧直接显示状态文案，点击整项触发打卡弹窗；
> 6. 我的页菜单改为 **3 项**（个人资料/检查更新/成就），无"关于"。
> 以截图为准，设计文档中与截图冲突处按本方案执行。

## 1. 实施总览

### 1.1 目标

按照设计文档的 MVVM 架构与 2 模块（`common` HAR + `entry` HAP）结构，从空白模板完整实现健康生活应用的 11 项功能（F1–F11）。

### 1.2 实施原则

| 原则 | 说明 |
|---|---|
| 架构先行 | 先搭模块骨架与目录结构，再填业务代码 |
| 自底向上 | Model → Data(Repository) → ViewModel → View，保证每层可独立编译验证 |
| 增量构建 | 每个任务完成后执行 `arkts_check` + `build_project`，确保不引入编译错误 |
| strict 合规 | 全程遵守 ArkTS strict 规则：禁 `any`/`as`/对象字面量类型/动态属性访问 |
| 资源复用 | UI 资源（media/string/color/float）已全部就绪于 `common`，直接 `$r` 引用 |
| 文档同步 | 任务完成后即时更新 `待完成功能列表.md` 状态 |

### 1.3 阶段划分

```
Phase 0  工程重构        common→HAR + 新建 entry HAP，打通依赖与构建
Phase 1  Model 层        枚举/常量/实体类（TaskInfo/TaskRecord/AchievementInfo/DayProgress）
Phase 2  Data 层         RdbHelper / PrefsHelper / TaskRepository 数据门面
Phase 3  ViewModel 层    Home/Task/Achievement/Mine/History ViewModel
Phase 4  基础页面骨架    Splash / MainTab(2 Tab 首页+我的) / HomeTab / MineTab 框架 + 导航
Phase 5  任务管理        TaskAddPage/TaskEditPage 独立页 + TaskListItem + ClockInDialog 打卡弹窗（F1/F2/F6）
Phase 6  连续天数与成就  streak 判定 + 成就解锁弹窗 + AchievementPage 成就页（F3/F4）
Phase 7  历史查看        首页 WeekCalendar 徽章周历（日期切换查看本周/历史）（F5）
Phase 8  服务卡片        FormAbility + WidgetCard1x2/2x2 + 卡片拉起/定时更新（F7–F10）
Phase 9  通知提醒        ReminderHelper + TimePicker/Toggle 接入（F11）
Phase 10 集成验证        全量构建 + 真机运行 + 联动回归
```

## 3. 依赖关系

```
Phase 0 (工程重构)
   ├─→ Phase 1 (Model) ─→ Phase 2 (Data) ─→ Phase 3 (ViewModel)
   │                                              │
   │                                              ↓
   │                                        Phase 4 (页面骨架)
   │                                              │
   │                    ┌─────────────────────────┼───────────────────┐
   │                    ↓                         ↓                   ↓
   │              Phase 5 (任务/打卡)      Phase 6 (成就)       Phase 7 (历史)
   │                    │                         │
   │                    └────────────┬────────────┘
   │                                 ↓
   │                          Phase 8 (卡片) ──→ Phase 9 (提醒)
   │                                                │
   └────────────────────────────────────────────────┘
                                                   ↓
                                           Phase 10 (集成验证)
```

- Phase 0 是所有后续阶段的前置。
- Phase 1→2→3→4 为纵向依赖链。
- Phase 5/6/7 可在 Phase 4 后并行（但建议按序，因 6 依赖 5 的打卡逻辑；6 的 P6-5 更新 Phase 4 产出的 AchievementPage 骨架）。
- Phase 8/9 相互独立，可并行。
- Phase 10 为最终收尾。

## 4. 风险与应对

| 风险 | 应对 |
|---|---|
| 模块拆分破坏现有构建 | Phase 0 完成后立即全量构建验证；逐步迁移而非一次性大改 |
| ArkTS strict 违规 | 每个文件编辑后 `arkts_check`；禁 any/as/对象字面量类型 |
| @Observed 数组刷新不生效 | 新增/删除整体替换数组引用；元素属性变更经 @ObjectLink |
| 卡片独立上下文不能读 AppStorage | FormAbility 直读 Repository，经 formProvider.updateForm 传 JSON |
| RDB 异步竞态 | 一律 async/await；Repository 单例串行化 |
| 跨天重置时机不确定 | onShow + onUpdateForm 双触发 maybeRollOverDate，幂等保护 |
| 截图与设计推断差异 | 以 screenshots/ 实际为准，编码时对照调整 |
| 打卡弹窗插画资源 | ClockInDialog 插画复用 ic_ad_bg.png 或任务专属图，编码时确认资源映射 |
| 任务页路由跳转 | TaskAddPage/TaskEditPage 用 router.push，完成后 router.back 返回首页刷新列表 |

## 5. 进度跟踪机制

- `待完成功能列表.md` 为唯一进度源：每个任务有 `状态`（待办/进行中/已完成）。
- 任务开始前标记 `进行中`，完成后标记 `已完成` 并在文件末尾"完成记录"追加一行（任务编号 + 完成日期 + 备注）。
- 若任务受阻，状态保持 `进行中` 并在"阻塞记录"追加说明。
